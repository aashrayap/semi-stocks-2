# Research: signal-desk-schema

## Flagged Items

- **`reviews` is not a stored object today.** No `review` artifact exists in `canonical/site-data/`, `canonical/40-engine/engine/site_data.py`, or `agents/state/`. The screenshot's `64 reviews` count is best explained as derived from `67 signals - 3 thesis_proposal_signal` records. Evidence: `canonical/site-data/build.json`, `canonical/site-data/signals.json`, `canonical/40-engine/engine/site_data.py`. Confidence: High for absence, Medium for screenshot-count inference.
- **`predictions` is overloaded.** Canonical site data has `claims.json` with 43 forward claims. Agent-side prediction YAMLs contain 26 actual prediction rows. The screenshot's `46 predictions` does not match agent predictions; it matches `43 claims + 3 pending thesis proposal signals`. Evidence: `canonical/site-data/claims.json`, `agents/state/predictions/*.yaml`, `canonical/site-data/signals.json`. Confidence: High for overload, Medium for screenshot-count inference.
- **Category aggregation currently spans two naming systems.** Thesis stages use display names like `N3 logic wafers`; company/signal data uses slugs like `n3_logic`; some source signals use values like `optical`. Evidence: `canonical/30-thesis/thesis.yaml`, `canonical/site-data/signals.json`, `canonical/site-data/companies.json`. Confidence: High.
- **`canonical/site-data/` is already the right app boundary, but still young.** It has a schema-first generator and artifacts, but the schema is shallow and does not yet expose a single Signal Desk view model. Evidence: `canonical/40-engine/engine/site_data.py`, `canonical/site-data/schema.json`, `docs/artifacts/lightweight-visual-layer/site-data-contract.md`. Confidence: High.
- **Wikiwise does not consume `canonical/site-data/`.** The external app still discovers `previews.json`, `search.json`, `graph.json`, and `map.json` under `wiki-site`; it copies and repairs a shell. Evidence: `/Users/ash/Documents/2026/wikiwise/Sources/Wikiwise/ExportBundle.swift`, `Compiler.swift`, `Resources/build.js`. Confidence: High.

## Findings

### Codebase

#### Q1 - Current Signal Desk data surface

**Answer:** The screenshot aligns with `canonical/site-data/`, generated by `canonical/40-engine/site_data.py` and `canonical/40-engine/engine/site_data.py`. The generated artifacts include `build`, `schema`, `pages`, `companies`, `signals`, `claims`, `entities`, `edges`, `thesis`, `reports`, `search`, and `graph`.

**Evidence:** `canonical/40-engine/engine/site_data.py`, `canonical/site-data/build.json`, `canonical/site-data/*.json`.

**Confidence:** High.

#### Q2 - Counts for reviews and predictions

**Answer:** Current generated counts are `40 companies`, `67 signals`, `43 claims`, and no review artifact. The screenshot shows `67 signals`, `64 reviews`, and `46 predictions`. The best fit is:

- `reviews = 64 categorized/reviewed signals`, excluding 3 pending thesis proposal signals.
- `predictions = 43 forward claims + 3 pending thesis proposal signals`.

This is an inference from counts, not an explicit schema contract.

**Evidence:** `canonical/site-data/build.json` reports 40 companies, 67 signals, 43 claims. `signals.json` has 3 `thesis_proposal_signal` rows. Agent prediction YAMLs have 26 prediction rows, so screenshot `46` is not agent-prediction count.

**Confidence:** Medium, because the screenshot UI source is not present in the repo.

#### Q3 - Wide-to-narrow code path

**Answer:** The current path is:

`10-wiki markdown + 20-data YAML + 30-thesis YAML + 40-engine synthesis -> canonical/site-data JSON -> future reader/UI`.

`canonical/wiki-site/` remains generated Wikiwise compatibility output. `canonical/50-reports/latest.html` is a published artifact, not the source for `site-data`.

**Evidence:** `README.md`, `docs/architecture.md`, `docs/artifacts/canonical-propagation-model/07_implementation-spec.md`, `canonical/40-engine/engine/site_data.py`.

**Confidence:** High.

#### Q4 - UI-required fields

**Answer:** The screenshot needs only:

- Category tab/table: `category_id`, label, status, period, company count, signal count, claim count, ticker list.
- Company tab/detail: `ticker`, name, status, category, positions/exposure, metrics, signal count, claim count, source page, thesis snippet.
- Signal tab/filter: `kind`, category, company/ticker, direction, evidence text, source path/page.
- Claim/proof gate list: ticker/company, category, claim text, status, verify target/window, source.
- Filters: category, signal kind or direction, search across ticker/category/signal text.

**Evidence:** Screenshot, `companies.json`, `signals.json`, `claims.json`, `thesis.json`.

**Confidence:** High for current UI shape; Medium for unseen source implementation.

#### Q5 - App view models versus canonical source

**Answer:** `canonical/site-data/` and `canonical/wiki-site/` are generated view/export models. Canonical source remains in `10-wiki`, `20-data`, `30-thesis`, and `50-reports`.

**Evidence:** `docs/architecture.md`, `docs/doc-contract.md`, `docs/artifacts/lightweight-visual-layer/site-data-contract.md`.

#### Q6 - Relevant local changes

### Patterns

#### Q1-Q2 - Entity naming and evidence lineage

**Answer:** Canonical source entities are file-backed and lane-owned. Generated app entities have stable IDs with type prefixes: `page:*`, `company:*`, `signal:*`, `claim:*`, `thesis-stage:*`, `bottleneck:*`.

Evidence lineage is preserved through `source_path`, `source_page`, `source_ref`, page paths, and typed edges such as `derived_from_source`, `supports_claim`, `contradicts_claim`, and `updates_thesis_stage`.

**Evidence:** `canonical/40-engine/engine/site_data.py`, `canonical/site-data/edges.json`, `canonical/site-data/schema.json`.

**Confidence:** High.

#### Q3 - Narrowest user-visible signal

**Answer:** The narrowest current signal is a row in `signals.json`: `id`, `kind`, category/bottleneck, company/ticker or stage/proposal reference, `direction`, `evidence`, and source trace. It is the right object for the UI signal list.

**Evidence:** `canonical/site-data/signals.json`, `_build_signals()` in `site_data.py`.

**Confidence:** High.

#### Q4 - Stable IDs, labels, counts, snippets

**Answer:** Stable IDs should remain machine IDs. Display labels belong on category/company/page records. Counts should be derived in the app view model, not hand-stored in source records. Narrative snippets should stay traceable to source page/body or evidence text.

**Evidence:** Current `companies.json` stores `signal_counts` and `claim_counts` as generated fields; `build.json` stores artifact counts; source YAML stores actual claims/signals.

**Confidence:** High.

#### Q5 - Validation pattern

**Answer:** Existing validation is in `validate_site_data()` and `schema.json`. If Signal Desk schema becomes executable, extend that path instead of adding a parallel validator.

**Evidence:** `canonical/40-engine/engine/site_data.py`, `docs/artifacts/lightweight-visual-layer/site-data-contract.md`.

**Confidence:** High.

### Cross-Ref

#### Q1-Q2 - Generated bundle authority

**Answer:** Docs and code agree that `canonical/wiki-site/` is generated output, but Wikiwise still repairs it and rebuilds graph/map from canonical markdown. `canonical/site-data/` is cleaner for Signal Desk because it is data-first and generated by the canonical engine.

**Evidence:** `07_implementation-spec.md`, `ExportBundle.swift`, `site_data.py`.

**Confidence:** High.

#### Q3 - Can reviews be modeled as evidence assessments?

**Answer:** Not as a separate entity yet. Current data only supports `review` as derived UI state on signals or claims. Canonical signals are already curated enough to display; thesis proposal signals are pending and can be treated as not reviewed/promoted.

**Evidence:** No review artifact; 3 proposal signals have `status: pending`; canonical/company/thesis signals are emitted without review status.

**Confidence:** Medium-High.

#### Q4 - Can predictions be modeled as forward-looking claims?

**Answer:** For the UI screenshot, yes, but call them `claims` or `proof gates`, not `predictions`. True predictions live under `agents/state/predictions/` and have a different lifecycle: generated before earnings, scored after earnings, sidecar-only until promoted.

**Evidence:** `canonical/site-data/claims.json`, `agents/autoagent/program.md`, `agents/state/predictions/*.yaml`, `docs/artifacts/daily/04_design.md`.

**Confidence:** High.

#### Q5-Q6 - Minimum schema and parked fields

**Answer:** Minimum schema is category, company, signal, claim, position/exposure, and derived counts. Park review workflow, agent prediction import, graph/map detail, rich pages, and scoring until the core desk needs them.

**Evidence:** Current screenshot, `site-data` artifacts, lightweight visual layer design.

**Confidence:** High.
